## 2. O Que Funcionou Bem

### Sistema de Citações [N]/[W-N]
Diferenciador #1 do produto implementado e validado. Citações Gold `[N]` para RAG (base curada Medway) e Silver `[W-N]` para web. `validate_citations()` remove marcadores sem fonte real. Bloqueio de concorrentes em 3 camadas funcionando.

### Code Review Adversarial como Rede de Segurança
42 issues encontrados e 42 corrigidos — taxa de 100% de resolução. Incluindo bugs críticos como race conditions em singletons, regex incorretos, e o bug arquitetural do bind_tools/fallback. **Decisão do time: manter e reforçar o review adversarial.**

### Padrão `get_model(tools=)` — Breakthrough Arquitetural
Descoberto na Story 2.3 que `bind_tools()` em `RunnableWithFallbacks` perdia o fallback LLM. Solução definitiva: `get_model(tools=)` faz bind em primary E fallback antes de `with_fallbacks()`. Padrão adotado por todas as stories subsequentes.

### Execução Paralela de Stories
Stories 2.1, 2.2 e 2.3 executadas em paralelo no Sprint 4. Gerou conflitos coordenáveis em `graph.py` e `orchestrate_llm.py`, mas o resultado foi eficiente — 3 tools entregues simultaneamente.

### Calculadoras Médicas — Design Elegante
10 calculadoras como funções Python puras (<1ms, zero custo, sem I/O). Design com ÚNICA tool + dispatch por nome evita poluição do context window do LLM.

### Crescimento de Testes
De ~206 para 422+ testes. Cada story adicionou entre 6 e 62 testes novos. Code reviews adicionaram testes extras consistentemente.

## 3. O Que Pode Melhorar

### 3.1 Action Items da Retro Anterior Ficaram Órfãos
**Severidade: CRÍTICA (processo)**

A retro do Epic 1 gerou 5 action items. Completamos 1, progredimos em 2, não endereçamos 2 (justamente os de prioridade ALTA e CRÍTICA). **Root cause identificado:** gap no processo — o SM não referencia a retro anterior ao criar stories, e o code review não recebe contexto da retro.

**Follow-through da Retro Epic 1:**

| # | Action Item | Status no Epic 2 |
|---|------------|-------------------|
| 1 | Corrigir 23 issues de Ruff lint | ⏳ Parcial |
| 2 | Testes de integração anti over-mocking | ❌ Não endereçado |
| 3 | Revisar error handlers silenciosos | ⏳ Parcial — reviews pegaram, mas padrão recorreu |
| 4 | Validar pipeline E2E com LLM real | ❓ Sem evidência |
| 5 | Configurar .env com credenciais de API | ✅ Completado |

**Solução acordada:**
1. Adicionar "Retro Watch Items" nos story files existentes
2. Code review recebe action items pendentes como contexto
3. Prep sprint entre epics para executar action items

### 3.2 Error Handling Insuficiente (Recorrente)
**Severidade: ALTA**

Apareceu em 3 de 6 stories (2.1, 2.2, 2.5). Mesmo padrão reportado na retro do Epic 1 — `except Exception` silenciosos, exceções raw vazando para o LLM, falta de validação de input. Code reviews corrigiram caso a caso, mas o padrão continua recorrente.

### 3.3 Bugs HIGH Consistentes em Todas as Stories
**Severidade: ALTA**

10 issues HIGH em 6 stories — cada story teve pelo menos 1 bug crítico. Incluindo: race condition no singleton Pinecone, regex que não capturava último resultado, cross-turn leak no collect_sources, fallback chain quebrada pelo bind_tools. **Decisão do time:** isso NÃO é problema — é o code review funcionando como deve. Manter vigilância.

### 3.4 Zero Validação E2E
**Severidade: ALTA**

3 Epics (1, 2, 4) implementados sem nenhum teste end-to-end real (webhook → grafo → LLM → resposta WhatsApp). Os 3 testes de integração pré-existentes continuam falhando por credenciais GCP. Risco de bugs sistêmicos não detectados.

### 3.5 Technical Debt Incorrido
**Severidade: MÉDIA**

- `drug_lookup` usa `icontains` com B-tree index (performance ruim em escala)
- Bulário API depende de scraping da ANVISA (frágil)
- Drug model criado na Story 2.6 (não na 2.4) — sequência de implementação diferiu do design
- `medical_calculator` retorna "não encontrada" sem fallback para LLM

## 4. Key Insights

1. **Code review adversarial é o melhor asset de qualidade** — não reduzir, reforçar
2. **Gap retro → stories é processo, não pessoas** — faltava mecanismo de feed-forward
3. **Execução paralela de stories funciona** — mas exige coordenação em arquivos compartilhados (graph.py)
4. **Padrão get_model(tools=) merece ADR** — descoberta arquitetural importante que afeta todo o projeto
5. **Validação E2E urgente** — empilhar código sem teste real é risco crescente
6. **Prep sprint entre epics é necessário** — débitos técnicos não resolvem sozinhos durante stories
