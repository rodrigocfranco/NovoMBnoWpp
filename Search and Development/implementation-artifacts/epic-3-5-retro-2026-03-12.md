## 3. O Que Pode Melhorar

### 3.1 Mocks Escondem Problemas Reais em Boundaries Externos
**Severidade: CRÍTICA**

Preocupação principal levantada por Rodrigo Franco. Análise dos 7 findings HIGH deste sprint revelou que **6 de 7 teriam passado despercebidos sem o code review adversarial** — os mocks estavam escondendo exatamente esses problemas:

| Finding HIGH | Mock escondia? |
|---|---|
| Caption `null` descartava mensagens (3.2) | SIM — mock retornava caption limpo |
| Phone logado inteiro — LGPD (5.1) | SIM — testes usam phone fake curto |
| Exception genérica crashava pipeline (3.2) | PARCIAL |
| Providers não liam ConfigService timeout (5.1) | SIM |
| persist com RetryPolicy dead (5.1) | SIM |
| Closure sem phone/message_id (5.2) | SIM |
| user_id ausente nos logs (5.2) | SIM |

**Root cause:** Testes unitários com mock validam lógica interna, mas NÃO validam integração real com APIs externas. Over-mocking foi resolvido na camada de infraestrutura (Redis/PG), mas boundaries externos (WhatsApp, Whisper, Pinecone) continuam 100% mockados.

**Decisão do time:** Configurar WhatsApp Business API e executar testes reais ANTES de iniciar Epic 6.

### 3.2 Qualidade dos Testes
**Severidade: ALTA**

Em TODAS as 3 stories revisadas, code review encontrou testes com defeitos:
- Teste tautológico que sempre passava (5.2)
- Params errados em teste de calculadora (5.1)
- Teste misleading que testava "text" em vez de "audio" (3.2)

Testes com bugs dão falsa confiança.

### 3.3 Code Review como Única Linha de Defesa para Bugs de Integração
**Severidade: ALTA**

26 findings em 3 reviews (7 HIGH). O code review adversarial é extremamente eficaz, mas é a ÚNICA camada que pega bugs de integração. Se um reviewer não perceber, o bug vai para produção.

### 3.4 Story 3.1 com Status Inconsistente
**Severidade: MÉDIA**

O arquivo da story diz `Status: review`, mas o sprint-status.yaml diz `done`. Discrepância a resolver.

### 3.5 Debt Técnico Pré-Existente
**Severidade: BAIXA**

- 1 teste pré-existente falhando (test_error_fallback PregelNode API change)
- 40 erros pré-existentes de ruff
- ADR do padrão get_model(tools=) pendente desde Epic 2

## 4. Key Insights

1. **Mocks são primeira camada, não última** — boundaries externos devem ter pelo menos um teste com API real
2. **Code review adversarial continua sendo o maior asset de qualidade** — 100% dos findings HIGH corrigidos
3. **MVP feature-complete é o momento ideal para validação real** — antes de empilhar mais features
4. **Retro Watch Items funcionam** — mecanismo de feed-forward validado (retro → stories → reviews)
5. **Over-mocking tem níveis** — resolvemos infra (Redis/PG), próximo nível é serviços externos (WhatsApp, Whisper, etc.)
6. **Stakeholder validation depende de testes reais** — não dá para fazer demo com screenshots de pytest
